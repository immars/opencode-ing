# memory系统重构

## 全局记忆:

`PEOPLE.md`，需要记录多个人的用户画像；每个人带一个飞书的用户id，名字。每次注意更新画像的时候，更新到正确的人。

`SOUL.md`，只有一个。

`CRON_SYS.md`，不再存在：代码化，通过`cron sys session` 调用 agent 帮忙压缩信息；得到回复以后代码直接写对应的文件

## per-session 记忆：

- L0, L1, L2：每个聊天记录自己的L0, L1, L2
- TASK.md, CRON.md: 每个聊天记录出现的任务和定时任务。
- USERS.md: 聊天会话中，发过言的用户ID列表。每行一个。单聊只有一行。如果是群聊，那么只记录跟自己交互过（@过）的用户的ID

# opencode session 管理策略：

1. 每个飞书聊天，对应一个opencode session。无论单聊还是群聊。

2. 建立这个opencode session的时候，注入的context 内容如下：
* SOUL.md
* 如果是单聊，或者群聊的USERS.md不超过5个人，则注入所有这些人对应的`PEOPLE.md`中的用户画像
* 这个会话自己的 L2(最近5条), L1(最近5条), L0(当天的最近的60条)

3. 当session中的agent有新的回复的时候，plugin根据session标题中的飞书聊天id，把消息发给对应的飞书聊天中

# 定时任务处理

- TASK.md 内容发到对应的会话中。"任务项，请检查执行进度或开始执行：" + 任务内容
- CRON.md 内容命中当前时间以后，发到对应的会话中：""
- CRON_SYS 原来由agent直接执行的压缩和写文件的任务，改为由代码来驱动；agent只负责summary的生成。Plugin负责在cron sys session中，告诉agent：把下列<detail></detail>标签中的信息进行压缩处理，原则是：用语简洁。描述事实。不超过500字。尽量包括关键信息。如果篇幅允许，尽量包括关键字，例如任务ID之类的信息。把回复用<summary></summary>标签围起来。<detail>xxxx</detail>
- 当agent返回消息了以后，plugin去负责更新相应的文件内容。

# 代码结构

以根目录为代码工程根目录的方式。更符合通行做法，减少踩坑

# context注入机制

不再是prompt的方式，这样会多发一个消息，可能会有副作用。而是采取更合适的方式。
